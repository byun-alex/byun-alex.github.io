Brain Dashboard

A local Jarvis for every project on my machine.

Chat with Claude, drop into a real embedded terminal, switch project context, and watch the token cost of every AI call in real time.

A Node.js web app with no framework: a plain HTTP server, a WebSocket bridge, and vanilla JS on the front. The embedded terminal is a genuine Claude REPL running in the browser through ConPTY and xterm.js, not a mock. It is the home base I actually start my day in.

  • Node.js, zero framework
  • WebSocket + ConPTY
  • xterm.js
  • cost observability
  • headless Claude orchestration

living system

What it actually took. The embedded terminal nearly broke me. Plain node-pty fails on Node 24 on Windows with a bare spawn EINVAL, and so does the usual prebuilt fork; finding that the Homebridge prebuilt was the one binding that works took a full evening of dead ends. The lesson that stuck: when a native module fails, the fix is rarely your code, it is picking the right build of someone else's.

Interactive Life Planner

A weekly planner that asks whether you actually did it.

Time-block your week, then answer for every block: done, or skipped? A brain-dump box turns three messy lines into a planned week using Claude, with a preview before anything is applied.

Most planners are joyless, so people stop opening them. This one has streaks, XP, a mood check-in and a follow-through percentage, because the resolution loop (did you do the thing you planned?) is the whole product. Zero-dependency Node server, vanilla frontend, JSON store on disk. The AI never writes to your week without an explicit Apply.

  • zero-dep Node
  • product thinking
  • headless claude -p integration
  • defensive JSON validation
  • engagement design

weekend hack

What it actually took. Built in two days, but the interesting hour was deciding what the product is. The insight from the original spec was that planning apps die at the follow-through step, so I built the did-you-do-it loop first and the pretty grid second. Screenshot-testing the UI headlessly caught two real layout bugs I would have shipped blind. Not open source yet; the code runs my actual week.

content-factory

One AI orchestrator, four tools, a finished video at the end.

Claude drives music generation, visuals and FFmpeg assembly through a single documented pipeline, with human approval gates exactly where taste matters.

An end-to-end engine for faceless ambient and lofi channels. One shared pipeline runs multiple channels, each with its own locked style guide, so adding a channel is configuration, not a rewrite. Every published asset is traceable through run logs and consistent naming.

  • MCP tool wiring
  • multi-tool orchestration
  • FFmpeg
  • human-in-the-loop design
  • process documentation

multi-week build

What it actually took. The hard part was resisting full automation. Early versions tried to automate the take-selection step and the output got worse, because taste is the one layer a pipeline cannot fake. The two approval gates are a design decision, not missing code. Wiring four tools that were never meant to talk to each other taught me more about integration failure modes than any tutorial.

claude-session-continuity

Memory for a tool that forgets everything.

Claude Code starts every session blank. These custom skills give it a durable cross-session memory, so "what was I doing?" takes ten seconds instead of ten minutes.

Four slash-command skills over one central Session Diary: /wrapup checkpoints a session into a structured handoff, /catchup rebuilds context next time, /catchup-all briefs across every project from a one-line-per-session rollup, /checkin is the accountability variant. A fixed entry schema is the contract between the session that writes and the session that reads.

  • Claude Code skills + hooks
  • schema as contract
  • context engineering
  • documentation systems

living system

What it actually took. This came from genuine pain: a dozen projects in one folder and every morning spent re-explaining myself to a goldfish. The design goal became a sentence I kept testing against: a future session with no memory should be productive in ten seconds. Getting the entry format right mattered more than any code, because the reader is an AI with no context and no patience. I run it hourly, every day.

job-ad-extractor

The reliability layer around an LLM call.

Messy job ad in, strict validated JSON out. Prompt versioning, schema validation and an eval harness turn "it seems to work" into a measured pass rate.

LLM steps fail quietly: prose instead of JSON, a dropped field, an invented enum, a salary band where max is below min. Every output here passes a pydantic gate or gets rejected and logged. An eval runner scores each prompt version against a hand-labelled golden set, and a regression check flags any case that passed before and fails now.

  • prompt versioning
  • structured-output validation
  • eval harness
  • pydantic
  • provider-agnostic client

weekend hack

What it actually took. Small on purpose. I built it to internalise one idea: the difference between an LLM call you wrote and an LLM call you can prove behaves. Writing the golden set by hand was tedious and it is exactly where the value lives, because "quality" became a number I can watch move between prompt versions instead of a vibe.

The systems behind the tools.

These run my actual life, so the content stays private. The architecture is public and written up as case studies.

Second Brain: a personal accountability engine

An Obsidian vault where a CLAUDE.md behaviour spec turns the assistant into an accountability coach. A daily /checkin skill reviews yesterday's commitments, forces dated ones for today, and files the record. Built to fix my own follow-through problem; used every day. The behaviour is the product, the notes are just its memory.

Morning-brief agent: an email that writes itself

A scheduled Claude Code cloud agent that reads my calendar, inbox, to-dos and the weather each morning and emails me a brief before the day starts, plus a strategic Sunday wrap. The whole behaviour is defined in two Markdown spec files: edit a bullet, the brief changes. No code to redeploy.

study-brain: an LLM-maintained study wiki

A study system with a strict division of labour: I curate sources and ask questions, the AI writes and maintains the entire wiki. Every practice and exam question gets answered and linked back to the concept it tests, written beginner-first so revision compounds instead of evaporating.

Telegram capture: ideas from my phone to my vault

A capture pipeline: message a Telegram bot from anywhere, a Cloudflare Worker webhook stores it in KV, and my local system ingests it into the right vault with an acknowledgement back. Built because ideas die in the gap between having them and being at the computer.
